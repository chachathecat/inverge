# Personal Concept Graph Durable Persistence Design

## Purpose

This document defines the future durable persistence boundary for the Personal Concept Graph before any Supabase migration, row-level security policy, or production write path is added. It is a design-only guardrail for the learner-facing Inverge scope: **감정평가사 1차** and **감정평가사 2차**.

The graph should remain an operational learning-signal layer for the core loop: input → diagnosis → tracking → prediction → recommendation → execution → retry/rewrite. It must not become an AI 채점기, official grading system, broad dashboard, public archive, or unsupported-exam model.

## Why persistence is needed

The current Personal Concept Graph implementation is helper-level and uses an in-memory/test repository adapter. Durable persistence will eventually be needed so that a learner's concept-level learning state can safely survive across sessions and support:

- Today Plan prioritization using stable per-user concept state.
- Spaced review that is opt-out rather than opt-in.
- Recovery tracking after wrong, confused, or rewrite-needed work.
- Calm operational recommendations that end in retry, rewrite, or scheduled review.
- Deletion/export handling for user-owned learning state.

Persistence is not needed to store raw learner inputs. The graph only needs compact metadata signals that describe a user's learning state for a subject/unit.

## What is intentionally not stored

The future graph tables must be **metadata-only**. They must not store raw OCR, problem, answer, source, official-answer, model-answer, grading, or instructor-feedback text.

Forbidden durable graph fields include, but are not limited to:

- `rawUserText`
- `rawOcrText`
- `rawAnswerText`
- `answerText`
- `problemText`
- `questionText`
- `copyrightedText`
- `originalText`
- `fullText`
- `sourceText`
- `officialAnswer`
- `modelAnswer`
- `scorePrediction`
- `instructorComment`

The graph must not persist copyrighted problem text, raw third-party source text, full learner answers, OCR transcripts, official model-answer claims, instructor comments, or score predictions. Any production implementation should reject these fields before persistence and include automated tests for the rejection behavior.

## Service data vs derived learning signal

### User-owned service data

Personal Concept Graph rows are user-owned service data because they drive the learner's private study operations. They should be treated as part of the learner account state and should be available for user export and deletion.

### Derived metadata signal

The graph values are derived learning signals, not source records. Examples include concept state, confidence band, last result category, counters, recommended next task type, and next due timestamp. These values are metadata summaries generated from execution signals and review activity.

### Boundary rule

A production write path may read transient learner input upstream to derive safe metadata, but the graph table may only receive the derived metadata. Raw problem, OCR, answer, copyrighted, official answer, model answer, score prediction, and instructor-comment fields must be blocked at the data-boundary layer.

## Proposed future table design

No Supabase migration is created in this PR. The following schema is a proposal for a future migration PR.

### `personal_concept_nodes`

| Column | Proposed type | Required | Notes |
| --- | --- | --- | --- |
| `id` | `uuid` | yes | Primary key, generated by database or guarded server function. |
| `user_id` | `uuid` | yes | Owner user id; must match authenticated user for learner operations. |
| `exam_mode` | `text` | yes | Allowed values: `first`, `second`. No other exam modes are learner-facing scope. |
| `subject_id` | `text` | yes | Metadata subject identifier from the 감정평가사 curriculum reference. |
| `unit_id` | `text` | yes | Metadata unit identifier from the 감정평가사 curriculum reference. |
| `state` | `text` | yes | Allowed values: `unknown`, `confused`, `wrong`, `recovering`, `stable`. |
| `confidence` | `text` | yes | Confidence band from derived execution signal; do not store raw evidence text. |
| `last_result` | `text` | yes | Derived result category such as done, wrong, unknown, or needs_rewrite. |
| `last_task_type` | `text` | yes | Metadata task type such as retrieval, rewrite, or O/X. |
| `wrong_count` | `integer` | yes | Non-negative count of wrong/confused events. |
| `recovery_count` | `integer` | yes | Non-negative count of recovery events. |
| `stable_count` | `integer` | yes | Non-negative count of stable confirmations. |
| `next_recommended_task_type` | `text` | yes | Metadata-only next action, ending in retrieval, retry, rewrite, or scheduled review. |
| `next_due_at` | `timestamptz` | yes | Next scheduled review/action time. |
| `updated_at` | `timestamptz` | yes | Last metadata update time. |
| `metadata_only` | `boolean` | yes | Must be `true`; defensive marker for boundary tests and audits. |
| `created_at` | `timestamptz` | optional | Creation timestamp, if useful for auditing. |
| `version` | `integer` | optional | Optimistic concurrency or schema-version marker. |
| `source_status` | `text` | optional | Metadata source status such as helper_derived or production_derived. |

### Keys and constraints

- Primary key: `id`.
- Unique key: `user_id + exam_mode + subject_id + unit_id`.
- `exam_mode` check: only `first` and `second`.
- `state` check: only `unknown`, `confused`, `wrong`, `recovering`, and `stable`.
- Count checks: `wrong_count`, `recovery_count`, and `stable_count` must be non-negative.
- Metadata check: `metadata_only` must be `true`.
- Recommended implementation should add an updated-at trigger in the future migration PR, not in this design PR.

## Row-level security assumptions

Do not implement RLS in this PR. The intended future RLS model is:

- User can select only own rows where `user_id = auth.uid()`.
- User can insert only own rows where `user_id = auth.uid()`.
- User can update only own rows where `user_id = auth.uid()`.
- User can delete only own rows where `user_id = auth.uid()`.
- Service role may operate only through guarded server functions that validate authenticated user intent, exam scope, metadata-only payloads, and tenant boundaries.
- No public read access.
- No instructor cross-tenant access.
- No global training corpus access to raw user data.
- Instructor-console surfaces, if any future documentation references them, must stay separate from learner graph routes and must not read learner graph rows across tenants.

## Deletion/export policy

- Personal Concept Graph rows are user-owned service data and should be included in account export as metadata records.
- Account deletion should delete the user's `personal_concept_nodes` rows, or make them inaccessible and queued for hard deletion according to the platform deletion workflow.
- Deletion must not leave orphaned graph rows connected to a deleted learner identity.
- Export should include metadata fields such as exam mode, subject/unit identifiers, state, counters, recommendation type, due date, and timestamps.
- Export must not include raw OCR/problem/answer text because those fields must never exist in graph tables.

## Retention policy

- Keep active learner graph metadata while the account is active and the learner has not requested deletion.
- Allow product-defined pruning of stale metadata-only rows after a documented inactive period, but never use pruning as a substitute for user-requested deletion.
- Keep pseudonymized aggregate analytics only as a future opt-in or research layer, separated from user-owned service data.
- Do not place raw third-party problem text, raw OCR transcripts, learner answer text, official answer text, model-answer text, score predictions, or instructor comments in a model-improvement corpus.
- Any future aggregate analytics must be pseudonymized, must not expose single-user rows, and must not reconstruct raw user or third-party text.

## Migration plan

This PR intentionally does not add a Supabase migration or production database writes. A future implementation should proceed in small, auditable PRs:

1. Add a migration for `personal_concept_nodes` with constraints, indexes, timestamps, and RLS disabled until policies are added in the same or immediately paired migration review.
2. Add RLS policies for own-row select/insert/update/delete and explicitly verify no public read access.
3. Add a guarded server repository that maps existing Personal Concept Graph node metadata into the table without accepting forbidden raw fields.
4. Add contract tests for rejected forbidden fields and allowed `exam_mode`/`state` values.
5. Add a read-through or adapter switch for Today Plan recommendations after the durable repository is covered by tests.
6. Add export/delete integration for user-owned graph metadata.
7. Run closed-beta learner-loop, taxonomy, build, and lint checks before enabling any production write path.

## Rollback plan

If a future persistence implementation causes access, data-boundary, or route stability issues:

- Disable the durable repository feature flag and fall back to the existing helper-level/in-memory behavior for tests or non-production execution.
- Stop writes through guarded server functions before changing schema.
- Keep existing rows intact while investigating unless a privacy boundary issue requires immediate deletion.
- If a migration must be reverted, first remove production write usage, then drop policies/functions/table in a separate reviewed migration.
- Verify that Today Plan still produces safe fallback recommendations and does not render a score-only endpoint.

## Future production implementation PR sequence

1. **Schema and RLS migration PR:** create `personal_concept_nodes`, constraints, indexes, timestamps, and RLS policies; do not wire learner writes yet.
2. **Repository contract PR:** add a Supabase-backed repository behind a feature flag with forbidden-field rejection and metadata-only DTOs.
3. **Learner read/write integration PR:** route execution learning signals into the guarded repository and read graph metadata for Today Plan.
4. **Lifecycle PR:** add export/delete handling and retention documentation updates.
5. **Closed-beta enablement PR:** enable durable graph persistence for a limited closed-beta cohort after learner-loop, taxonomy, build, lint, and smoke checks pass.
